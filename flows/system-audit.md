# EngHub — аудит сценариев, процессов и пробелов

Дата: 2026-04-26
Источники: реальный код в `enghub-main/src`, `enghub-main/api`, `supabase/migrations`, `supabase/functions`. Везде, где приведена строка кода, ссылка ведёт на конкретный файл и место.

Документ отвечает на три вопроса:
1. Какие сценарии у платформы — от прихода нового проекта до выпуска чертежа в трансмиттале.
2. Как реально устроен день инженера, руководителя отдела и ГИПа — что они видят, что нажимают, какие переходы возможны, какие невозможны.
3. Что в процессе сломано, недоделано или отсутствует, и какие фичи добавить, чтобы каждый переход был осмысленным.

## 0. Карта системы за 30 секунд

EngHub — это четыре связанных контура, которые крутятся вокруг проекта:

1. **Управление работой** — проекты → задачи (Kanban с состояниями `todo → inprogress → review_lead → review_gip → done` и веткой `revision`).
2. **Документооборот** — чертежи → ревизии (R0/R1/R2…) → замечания (severity critical/major/minor) → трансмитталы (комплект для выпуска).
3. **AI-контур** — Copilot-оркестратор с девятью интентами (анализ проекта, декомпозиция задач, нормоконтроль, риск-форкаст и т.д.) + RAG по нормативке (`pgvector`) + защищённый apply через `validateApplyAction`.
4. **Окружающие модули** — совещания с PDF-протоколом, ConferenceRoom (WebRTC, реально живой), табель `time_entries`, дашборд ГИПа (Risk, S-curve, RACI), Telegram-бот, инженерные расчёты с экспортом в `.docx`.

Роли — четыре: `admin`, `gip`, `lead`, `engineer`. Привязка к отделу — через `app_users.dept_id`.

### 1.3 AI-разрешения (защищённый apply-контур)

`enghub-main/src/copilot/validateApplyAction.ts` — единственное место, где роли реально применяются для действий через Copilot:

- строки 44-51: жёсткий блек-лист действий, которые AI не имеет права делать ни для кого: `update_task`, `delete_entity`, `assign_user`. То есть Copilot не может «удалить задачу» или «переназначить исполнителя» — это всегда ручное действие пользователя.
- строки 56-58: инженер не может через AI создавать задачи.
- строки 104-106: AI не может в payload подсунуть `assigned_to` — назначение исполнителя всегда ручное.
- лимит 50 элементов в одной apply-операции (защита от случайной массовой генерации).

Хорошее место — отделение «инсайта» от «действия»: оркестратор сначала кладёт `ai_actions` в статусе `pending`, и только после явного подтверждения пользователя выполняется реальный INSERT/UPDATE.

### 1.4 Дыры в RBAC

| Проблема | Серьёзность | Что значит на практике |
|---|---|---|
| RLS не различает роли | критично | Если кто-то достанет `anon_key` или авторизуется как любой `authenticated` пользователь — увидит и сможет писать в чужие проекты. Все ролевые проверки только во фронте. |
| `admin` определяется по email-строке | высокая | Замена email в auth → эскалация до admin. |
| Инженер может создать чертёж/ревизию/трансмиттал | высокая | Никаких guard'ов в `DrawingsPanel`, `RevisionsTab`, `TransmittalsTab` — только UI-кнопки видно или нет, но API открыт. |
| Любой может закрыть чужое замечание | средняя | `changeReviewStatus` без проверки на автора/назначенного. |
| Lead вне своего отдела | средняя | UI обычно скрывает, но если задача попала в выборку — поменяет приоритет/назначение. |

**Что добавить, минимально и быстро:**

1. RLS-политики на основе подзапроса в `app_users.role`/`dept_id`. Базовый шаблон:
   ```sql
   create policy "tasks_lead_dept" on tasks for select using (
     exists (select 1 from app_users u where u.id = auth.uid() and (
       u.role = 'gip' or
       (u.role = 'lead' and u.dept_id = tasks.dept_id) or
       (u.role = 'engineer' and tasks.assigned_to = u.id)
     ))
   );
   ```
2. Серверная валидация во всех API-роутах в `enghub-main/api/*` — проверять role/dept_id через service-key и контекст пользователя из JWT.
3. Audit-лог смены ролей: `role_audit(user_id, old_role, new_role, changed_by, changed_at)` + триггер.
4. Делегирование на отпуск: `role_delegation(from, to, role, valid_until)` — чтобы лид мог временно передать права без правки `app_users`.
5. Роль `observer` — только SELECT, без write — для заказчика и эксперта-консультанта.

## 2. Жизненный цикл проекта

### 2.1 Создание

- Кнопка только у ГИП — `App.tsx:1683`. Поля: название, шифр, дедлайн, отделы.
- При создании проект сразу пустой: ноль задач, ноль чертежей. ГИП должен дальше «насыпать» структуру.
- Нет автоматического создания скелета задач по шаблону раздела (АР/КМ/ВК/ОВ/ЭО). Это узкое место — каждый проект собирается с нуля.

### 2.2 Архив

- Архивирование — `App.tsx:2094`, тоже только ГИП. Помеченный архивный проект исчезает из списка активных.

### 2.3 Назначение второго ГИПа / разделение проектов

В коде у проекта только один владелец-ГИП. Сценарий «один проект ведёт ГИП-1, другой проект — ГИП-2» работает так: каждый ГИП видит **все проекты** (RLS не разграничивает), но фактически работает только с тем, который сам создал, потому что только он назначает задачи. Это слабое место — если в институте несколько ГИПов, фильтрации «мои проекты» в UI нет.

**Что добавить:**
- Поле `projects.gip_id` (или связка m2m, если ГИПов несколько на проект).
- Фильтр «мои проекты» на главной.
- При назначении нового ГИПа — пуш ему дайджеста проекта.

        │                          └──────┬──────┘                      └────┬─────┘
        │                                 │ лид: "Утвердить → ГИПу"           │ инженер: "Снова в работу"

**Места перехода в коде:**
- `App.tsx:998-1051` — `updateTaskStatus()` (общий обработчик)
- `App.tsx:1136-1141` — `assignTask()` (назначение инженеру лидом)
- `App.tsx:1142-1165` — `issueRevision()` (ГИП выпускает следующую ревизию задачи — фактически создаётся новая задача с `parent_task_id` и `revision_num + 1`)
- `App.tsx:1225-1240` — `getTaskActions()`: список доступных кнопок для текущей роли и текущего статуса. Это «правда» о том, кто что может в каком состоянии.

### 3.2 Что не реализовано в lifecycle

| Чего нет | Последствие |
|---|---|
| SLA-таймера и авто-эскалации | Никто не получит автопинг при приближении дедлайна |
| Авто-напоминания за день/час | Дедлайн просто молча пройдёт |
| Возврат `done → todo` | Если задачу повторно открыли — обходом через ручное создание |
| Зависимости `blocks/blocked_by` | Только `parent_task_id` для ревизий |
| Промежуточных вех | Один `deadline` на всю задачу, чекпоинты выпуска не отслеживаются |
| Email-канала уведомлений | Только in-app `notifications` + Telegram |
| Частичная приёмка (50% готово) | Чёрно-белое: либо `done`, либо нет |
| Триггера `task_history` в БД | История пишется приложением — если код забыл, истории нет |

### 3.3 Реальный сценарий «понедельник 09:00 — пятница 17:00»

Сценарий **по реально существующему коду**, без фантазий:

**Понедельник 09:00. ГИП Соколов получил новый проект «ТЭЦ-Восток, реконструкция».**

- Открывает EngHub → жмёт «Новый проект». Заполняет шифр, дедлайн 12.06, отделы (КМ, ОВ, ЭО, АР).
- Кнопка «+ Новая задача» → форма (`App.tsx:1313-1359`). Создаёт задачу «КМ-001 Расчёт фундамента», дедлайн пт 17:00, **назначает руководителю отдела КМ** (выпадашка `НАЗНАЧИТЬ РУКОВОДИТЕЛЮ`, строка 1325).
- В БД: `INSERT INTO tasks` со статусом `todo`, `assigned_to = lead_km.id`. Срабатывает `createNotification()` (`App.tsx:948-955`) — лид КМ получает in-app уведомление.

**Понедельник 10:30. Лид КМ (Юлия) открывает приложение.**

- В Kanban-колонке `todo` видит новую красную карточку. Открывает деталь.
- Жмёт «Назначить инженеру» (`App.tsx:1424-1427`, видна только лиду). Из выпадашки выбирает инженера КМ Петрова. `assignTask()` → `PATCH tasks { assigned_to: petrov.id, status: 'todo' }`. Петров получает уведомление.

**Понедельник 11:00. Инженер Петров.**

- Видит карточку «КМ-001» в своём `todo`. Жмёт «▶ Взять в работу» — единственная доступная ему кнопка по `getTaskActions` (`App.tsx:1234`). Статус → `inprogress`. История пишется записью «status: todo → inprogress».
- Параллельно создаёт чертёж в `DrawingsPanel`: код «КМ-001», лист 1, формат A1, ревизия R0, статус `draft`. Загружает PDF.

**Понедельник–среда. Работа.**

- Петров двигает задачу мышью: статус не меняет, просто работает с файлом. Иногда жмёт `Записать часы` — `time_entries` (`TimelogPanel.tsx`).

**Среда 14:00. Петров завершил.**

- Жмёт «↑ Отправить на проверку» — статус `review_lead`. Лид получает уведомление.

**Среда 15:00. Юлия (лид) проверяет.**

- Открывает деталь задачи, видит чертёж и пояснительную записку. Запускает Copilot: «Нормоконтроль КМ-001» — Copilot вызывает `analyze_drawing` через Claude Vision (`api/orchestrator.js`, интент `analyze_drawing`). Получает чек-лист потенциальных замечаний.
- Решает: 3 critical + 1 minor. Открывает вкладку «Замечания», создаёт три записи `severity: critical` через `submitReview` (`App.tsx:1069-1081`).
- Жмёт «✗ На доработку» в детали задачи. Статус → `revision`. В БД пишется `status_change` с комментарием. Петров получает уведомление с комментарием.

**Четверг 09:00. Петров.**

- Видит задачу в `revision`. Читает три замечания критичные. Правит чертёж. **Нет встроенного diff-инструмента — сам сравнивает старую и новую версию визуально.**
- Создаёт новую ревизию чертежа: `App.tsx:1089-1102` → запись в `revisions` с `from_revision='R0', to_revision='R1'`. Файл перезагружается в Storage поверх или как отдельный объект (см. п. 4.5 ниже — это вообще-то проблема, возможно перезаписывается).
- Меняет статусы трёх замечаний на `in_progress` → потом `resolved` после самопроверки.
- Жмёт «▶ Снова в работу» → `inprogress`, и через час «↑ Отправить» → `review_lead`.

**Четверг 14:00. Лид принимает.**

- Жмёт «✓ Утвердить → ГИПу». Статус `review_gip`. ГИП получает уведомление.

**Пятница 10:00. ГИП Соколов.**

- Открывает задачу. Запускает Copilot: «compliance_check» по тексту ПЗ — RAG-поиск по нормативке, выдаёт чек-лист соответствия СП.
- Видит, что замечаний больше нет. Жмёт «✓ Завершить». Статус `done`.

**Пятница 11:00. Чертёж в трансмиттал.**

- ГИП открывает «Трансмитталы», создаёт новый: `App.tsx:1103-1114`. Номер генерируется как `TR-{projectCode}-{seq}`. Добавляет позиции — drawing «КМ-001», ревизия R1.
- Меняет статус `draft → issued`. Жмёт «🖨 Экспорт PDF» — `utils/export.ts:111-159`, открывается окно `window.print()` с HTML-таблицей.
- **Дальше — вручную:** скачать PDF, отправить заказчику в почту/Telegram. **Авто-рассылки нет.**

### 3.4 Идеи по улучшению цикла задач

1. **SLA-метрики и авто-эскалация.** Триггер на `tasks.deadline`: за 24 часа до дедлайна — пинг исполнителю и лиду; при просрочке — пинг ГИПу + флажок `overdue` на дашборде.
2. **Контрольные точки.** Добавить `task_milestones[]` (название, дата, статус). На задачу «Расчёт» — три вехи: эскиз / уточнение / финал. Видны на Gantt.
3. **Зависимости.** Таблица `task_dependencies(task_id, blocks_id)`. На уровне UI — на детали задачи список «эта задача блокирует / заблокирована». При `done` — авто-переход блокируемых из `blocked` в `todo`.
4. **Email-канал поверх `createNotification`.** Один edge-function `send-notification`, который синхронно пишет в `notifications`, асинхронно шлёт email (через Resend/SendGrid) и Telegram-DM. Шаблоны: «Назначена», «Возврат на доработку», «Близкий дедлайн», «Просрочка».
5. **Перевод в `revision` обязательно с комментарием.** В `App.tsx:1234-1238` сейчас можно вернуть на доработку без обязательного текста — лид иногда возвращает молча, инженер не понимает что править. Добавить required-поле `comment`.

**Среда 14:30. Лид Юлия** открывает «Замечания», создаёт три critical: «Дверь проходит сквозь стену», «Окно на отметке -0.500», «Размерная цепочка не замкнута».

**Пятница 09:00. Юлия** проверяет резолюции, ставит чертёжу `status=approved`.

**Пятница 11:00. ГИП Владимир** создаёт трансмиттал TR-ТЭЦ-001, добавляет позицию (АР-101, R1). Меняет статус `draft → issued`. Жмёт «🖨 PDF» — открывается окно печати, выгружает PDF вручную, отправляет в почту заказчику.

### 4.4 Что отсутствует в документообороте

1. **Цифровая подпись.** Никакой криптографии — трансмиттал просто PDF с печатью браузера.
2. **Diff PDF между ревизиями.** Инженер сравнивает версии глазами.
3. **Авто-проверка штампа на соответствие ГОСТ 21.501.** В штампе должны быть: марка раздела, исполнитель, проверяющий, ГИП, дата, шифр. Сейчас никто это не валидирует.
4. **Email/EDM при выпуске.** Status → issued ничего никому не шлёт, ГИП руками ходит в почту.
5. **Реестр исходящих с годовой нумерацией.** Сейчас номер `TR-{projectCode}-{seq}` локален к проекту. Нет общеинститутского регистра «Out-2026-042» с уникальностью.
6. **Версионирование файлов.** Нужно проверить — в Storage один файл перезаписывается или каждая ревизия лежит отдельно. По коду в `createProjectDrawing` видно, что путь не включает revision, то есть скорее всего **R0 затирается R1**. Это серьёзный риск для аудита.
7. **Блокировка выпуска при открытых critical.** Сейчас никто не мешает ГИПу выпустить трансмиттал, пока есть `severity=critical, status=open`. Логика «critical блокирует issue» только в README, в коде не нашёл.

### 4.5 «Фишечки» для документооборота

1. **AI-извлечение замечаний из PDF-комментариев.** Заказчик присылает PDF с pin-комментариями — Copilot парсит pdf.js, создаёт `reviews` с координатами на листе. Раздел DrawingsPanel уже умеет в Vision (`analyze_drawing`), стек стоит.
2. **Авто-валидация штампа.** На переходе `draft → in_work` — Vision-проверка наличия в штампе обязательных полей по ГОСТ 21.501. Если поле пустое — красный флаг и блок до заполнения.
3. **Side-by-side diff PDF.** Pdf.js или pdf-diff. На вкладке «Ревизии» — две колонки, выделение изменений жёлтым/красным.
4. **Авто-нормоконтроль комплектности.** На статус `issued` транзитом проверять: для АР должны быть планы всех этажей + фасады + разрезы; для ВК — план разводки + аксонометрия + спецификация. Если чего-то нет — warning-модалка перед выпуском.
5. **Check-in/check-out (как PDM).** `drawings.locked_by`, `locked_until`. Инженер «берёт на редакцию» — на 4 часа никто другой не редактирует. На выпуск ревизии — авто-unlock.
6. **Регистр исходящих с QR.** Глобальная таблица `outgoing_registry` с авто-номером по году. В PDF трансмиттала встраивается QR со ссылкой на запись — заказчик сканирует и подтверждает получение.
7. **Хранение файла per-revision.** Путь в Storage `drawings/{project}/{drawing_code}/{revision}.pdf`. Старые ревизии — read-only. Это критично для арбитража.
8. **Блокировка `issued` при открытых critical.** В `changeTransmittalStatus` (`App.tsx:1115`) добавить проверку: если в составе позиции есть чертёж с `reviews.status='open' AND severity='critical'` — отказ.

### 5.3 Apply-цикл

1. Copilot отрабатывает интент, готовит payload.
2. INSERT в `ai_actions` со статусом `pending`.
3. UI показывает карточку «Создать 5 задач» / «Создать 3 замечания».
4. Пользователь жмёт «Подтвердить». `applyAction(action, true)`.
5. `validateApplyAction` пропускает или режет.
6. INSERT в реальные таблицы. `ai_actions.status = approved`.

### 5.12 Идеи фич AI-слоя

7. **AI-черновик ТЗ из брифа.** Заказчик присылает 1 страницу — `smart_decompose + norm_control` собирают черновое ТЗ с разделами и ГОСТ-ссылками.

## 7. Сводка пробелов

### Критичные

1. **RLS не различает роли.** Все политики на `authenticated`. На уровне БД защиты нет.
