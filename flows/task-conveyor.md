## 1. ОБЩАЯ ЛОГИКА

### Что происходит

Инженерная задача проходит следующий путь:

```
ГИП создаёт задачу
        ↓
Инженер выполняет
        ↓
Отправляет на проверку Lead'у
        ↓
Lead проверяет (может вернуть)
        ↓
Если OK → отправляет на ГИП
        ↓
ГИП утверждает или возвращает
        ↓
Если утверждено → задача выпускается
```

### Ключевые принципы

- **Инженер ВСЕГДА ответственный** — задача не может быть без ответственного
- **Проверки в два этапа:** на уровне отдела (Lead) и уровне проекта (ГИП)
- **Возврат ВСЕГДА с причиной** — нельзя вернуть без пояснения
- **Блокировки видны:** если задача ждёт данных от другого отдела, статус отражает это
- **Парт-статусы:** при замечаниях/рецензиях задача не переходит на следующий этап

## 2. РОЛИ В СИСТЕМЕ

### ГИП (Главный инженер проекта)

**Ответственность:**
- Приём, распределение, утверждение задач по проекту
- Контроль сроков и качества всего проекта
- Принятие решения по замечаниям (может запросить переделку у Lead)
- Выпуск результатов (ревизии, трансмитталы)

**Действия в системе:**
- Создаёт задачу
- Просматривает статус всех задач проекта
- Отправляет задачу в отделы (назначает Lead)
- Проверяет результат от Lead (читает замечания, просматривает чертежи)
- Утверждает (→ статус "Утверждено") или возвращает (→ "На доработке" + причина)
- Выпускает ревизии и трансмитталы
- Видит все задачи и все замечания в своём проекте

**RLS в системе:**
- Видит задачи только своего проекта
- Может менять статус задачи на любой (кроме некоторых — см. таблицу стадий)
- Может добавлять замечания к любому результату

### Lead отдела (Начальник отдела)

**Ответственность:**
- Проверка качества работы инженеров своего отдела
- Контроль, чтобы инженер выполнил требования ГИПа
- Выявление ошибок и недоработок (замечания)
- Отправка результата на проверку ГИПу (только если качество OK)

**Действия в системе:**
- Видит только задачи, назначенные в его отдел
- Видит список инженеров в своём отделе
- Проверяет статус выполнения (когда инженер отправляет "На проверке")
- Добавляет замечания (с указанием критичности и места в документе)
- Принимает результат (→ "На проверке ГИПу") или возвращает (→ "На доработке" + причина)
- Видит сроки
- Может просить время у ГИПа (блокировка + причина)

**RLS в системе:**
- Видит задачи только своего отдела (по team_id)
- Может менять статус на "На проверке ГИПу" и "На доработке"
- Может добавлять замечания

### Инженер

**Ответственность:**
- Выполнение работы в срок и в соответствии с требованиями
- Подготовка результатов (чертежи, расчёты, документы)
- Исправление замечаний от Lead

**Действия в системе:**
- Видит свои назначенные задачи
- Устанавливает статус "В работе", когда начинает
- Прикрепляет результаты (чертежи, файлы)
- Отправляет на проверку (→ "На проверке Lead'у")
- Получает замечания от Lead
- Возвращается в "На доработке" и исправляет
- Видит финальный результат (утверждено или нет)

**RLS в системе:**
- Видит только свои задачи (auth.uid() = assignee_id)
- Может менять статус только на "В работе" и "На проверке Lead'у"
- Может прикреплять файлы к своей задаче

## 3. СТАДИИ ЗАДАЧИ

Каждая задача проходит через строго определённые стадии. Переход между стадиями возможен только при выполнении условий.

### Таблица стадий

| № | **СТАДИЯ** | **КТО ОТВЕТСТВЕННЫЙ** | **ЧТО МОЖНО ДЕЛАТЬ** | **КТО МОЖЕТ ПЕРЕВЕСТИ** | **КУДА ДАЛЬШЕ** |
|---|---|---|---|---|---|
| 1 | **Создана** | Инженер (назначен ГИПом) | Ничего, только читать | ГИП | "В работе" |
| 2 | **В работе** | Инженер | Прикреплять файлы, обновлять прогресс | Инженер | "На проверке Lead'у" |
| 3 | **На проверке Lead'у** | Lead | Читать, добавлять замечания | Lead / Инженер | "На доработке" или "На проверке ГИПу" |
| 4 | **На проверке ГИПу** | ГИП | Читать, добавлять замечания | ГИП | "Утверждено" или "На доработке" |
| 5 | **На доработке** | Инженер (снова) | Исправлять, прикреплять новые файлы | Инженер | "На проверке Lead'у" |
| 6 | **Ожидание данных** | (система) | Статус "заморожена", видима в списке | Система (автоматически при связи) | "В работе" (при получении данных) |
| 7 | **Утверждено** | ГИП (финально) | Только читать, результат закреплён | ГИП | "Архив" (конец) |

### Правила переходов

```
Создана 
  ├─→ (ГИП только) "В работе"
  └─→ "Ожидание данных" (если зависит от другой задачи)

В работе
  ├─→ (Инженер) "На проверке Lead'у"
  └─→ "Ожидание данных" (если срочно нужны данные)

На проверке Lead'у
  ├─→ (Lead) "На доработке" ← если есть замечания
  ├─→ (Lead) "На проверке ГИПу" ← если качество OK
  └─→ (Инженер вернулся) "В работе" ← откатить свою отправку

На проверке ГИПу
  ├─→ (ГИП) "Утверждено" ← результат финализирован
  ├─→ (ГИП) "На доработке" ← замечания важные
  └─→ (Инженер) "В работе" ← откатить

На доработке
  └─→ (Инженер) "На проверке Lead'у" ← готов результат 2-й версии

Ожидание данных
  ├─→ (ГИП вручную) "В работе" ← данные пришли
  └─→ (система автоматически) "В работе" ← если связанная задача утверждена

Утверждено
  └─→ (только для истории) "Архив"
```

## 4. ДЕЙСТВИЯ (ОПЕРАЦИИ)

### Отправить на проверку

**Кто может:** Инженер (только из "В работе")

**Условие:**
- Задача в статусе "В работе"
- Есть хотя бы один файл прикреплён (чертёж или результат)

**Что происходит:**
1. Статус → "На проверке Lead'у"
2. Lead получает уведомление в Telegram и систему
3. Инженер видит индикатор "на проверке"
4. Сроки начинают отсчитываться для Lead

**REST API:**
```typescript
PATCH /api/tasks/{task_id}
{
  "status": "review_lead",
  "updated_at": "2026-05-05T12:30:00Z"
}
```

---

### Добавить замечание

**Кто может:** Lead (из "На проверке Lead'у") или ГИП (из "На проверке ГИПу")

**Параметры замечания:**
- `severity`: "minor" | "major" | "blocker"
- `location`: лист, узел или ось (например, "Лист 2, узел A3")
- `text`: описание проблемы
- `tag`: "dimensioning" | "consistency" | "calculation" | "compliance" | "other"

**Что происходит:**
1. Замечание сохраняется в таблице `reviews`
2. Инженер видит замечание и место расположения
3. Задача остаётся в "На проверке" до решения
4. Количество замечаний показывает рецензионный статус

**REST API:**
```typescript
POST /api/tasks/{task_id}/reviews
{
  "severity": "major",
  "location": "Лист 3, узел B2",
  "text": "Сечение кабеля не соответствует расчёту",
  "tag": "calculation"
}
```

---

### Принять результат (Lead → ГИП)

**Кто может:** Lead (только из "На проверке Lead'у")

**Условие:**
- Нет замечаний со статусом `resolved = false`
- Или все замечания marked как resolved Lead'ом

**Что происходит:**
1. Статус → "На проверке ГИПу"
2. ГИП получает уведомление
3. Lead закрывает проверку
4. Инженер видит "на проверке у ГИПа"

**REST API:**
```typescript
PATCH /api/tasks/{task_id}
{
  "status": "review_gip",
  "review_completed_by_lead_at": "2026-05-05T13:00:00Z"
}
```

---

### Вернуть на доработку (с причиной)

**Кто может:** Lead (из "На проверке Lead'у") или ГИП (из "На проверке ГИПу")

**Обязательно:**
- Добавить минимум одно замечание со статусом `resolved = false`
- Указать приоритет возврата (minor/major/blocker)

**Что происходит:**
1. Статус → "На доработке"
2. Инженер получает уведомление с причинами
3. Замечания видны инженеру с указанием критичности
4. Счётчик попыток увеличивается
5. Новый срок рассчитывается от даты возврата

**REST API:**
```typescript
PATCH /api/tasks/{task_id}
{
  "status": "rework",
  "returned_by": "lead" | "gip",
  "reason": "Найдены критические ошибки в расчётах",
  "reviews": [
    { "id": "rev_123", "resolved": false }
  ]
}
```

---

### Утвердить результат (ГИП финально)

**Кто может:** Только ГИП (из "На проверке ГИПу")

**Условие:**
- Все замечания разрешены
- Или ГИП считает результат приемлемым несмотря на замечания

**Что происходит:**
1. Статус → "Утверждено"
2. Результат становится частью выпуска (ревизии)
3. Инженер видит зелёный статус
4. Документ готов к передаче в трансмиттал
5. История фиксируется с датой и подписью ГИПа

**REST API:**
```typescript
PATCH /api/tasks/{task_id}
{
  "status": "approved",
  "approved_by_gip_at": "2026-05-05T14:00:00Z",
  "approved_by_gip_id": "user_gip_123"
}
```

---

### Создать связанную задачу (межотдельная зависимость)

**Кто может:** ГИП (когда создаёт основную задачу)

**Параметры:**
- `parent_task_id`: ID основной задачи
- `required_from_team_id`: ID отдела, который должен предоставить данные
- `required_data_description`: что именно нужно (например, "Данные по теплоустойчивости кабеля")
- `deadline_for_dependency`: когда нужны данные (раньше, чем дедлайн основной)

**Что происходит:**
1. Создаётся связь в таблице `task_dependencies`
2. Основная задача → "Ожидание данных"
3. В отделе-исполнителе создаётся новая задача (или привязывается существующая)
4. Зависимая задача показывает, что результат требуется другой командой
5. Когда зависимая задача утверждена → основная автоматически → "В работе"

**REST API:**
```typescript
POST /api/tasks/{task_id}/dependencies
{
  "required_from_team_id": "team_ks_123",
  "required_data_description": "Схема электроснабжения объекта",
  "deadline_for_dependency": "2026-05-10T17:00:00Z",
  "related_task_id": "task_related_456"  // если есть готовая задача
}
```

---

### Разблокировать задачу (получить данные)

**Кто может:** Система (автоматически) или ГИП (вручную)

**Автоматический механизм:**
- Когда зависимая задача переходит в "Утверждено"
- Основная задача получает сигнал: `dependency_resolved = true`
- Система меняет статус с "Ожидание данных" на "В работе"
- Инженер получает уведомление "Данные готовы, можно продолжать"

**Ручной механизм:**
- ГИП видит задачу в "Ожидание данных"
- Может вручную перевести в "В работе" если данные поступили через другой канал
- Добавляет комментарий "Данные поступили из [источник]"

**REST API:**
```typescript
PATCH /api/tasks/{task_id}
{
  "status": "in_progress",
  "dependency_resolved": true,
  "unblocked_by_gip_at": "2026-05-05T15:00:00Z",
  "comment": "Получены данные от КЖ отдела"
}
```

## 5. МЕЖОТДЕЛЬНЫЕ СВЯЗИ

### Механизм создания зависимости

```
ГИП видит задачу
        ↓
Понимает: нужны данные от другого отдела
        ↓
Создаёт "связанную задачу" (task_dependencies)
        ↓
В отделе появляется новая задача
        ↓
Lead видит "это нужно для другого отдела"
        ↓
Инженер выполняет и отправляет на проверку
        ↓
Lead проверяет и отправляет ГИПу
        ↓
ГИП утверждает
        ↓
Система автоматически РАЗБЛОКИРУЕТ основную задачу
        ↓
Инженер основной задачи получает уведомление
        ↓
Может продолжить работу
```

### Хронология

| Шаг | События | Статусы |
|---|---|---|
| T0 | ГИП создаёт основную задачу | Основная: "Ожидание данных" |
| T0 | ГИП добавляет зависимость | Зависимая: "Создана" (в другом отделе) |
| T1-T2 | Инженер другого отдела выполняет | Зависимая: "В работе" |
| T3 | Инженер отправляет на проверку Lead'у | Зависимая: "На проверке Lead'у" |
| T4 | Lead проверяет и отправляет ГИПу | Зависимая: "На проверке ГИПу" |
| T5 | ГИП утверждает | Зависимая: "Утверждено" |
| T5 | **Система автоматически** разблокирует | Основная: "В работе" (из "Ожидание данных") |
| T5+ | Инженер основной задачи может продолжить | Основная: "В работе" |

### Возврат и повторная доставка

Если при проверке зависимую задачу вернули на доработку:

```
Зависимая: "На доработке"
        ↓
Инженер исправляет
        ↓
Отправляет снова Lead'у
        ↓
Lead проверяет → ГИПу
        ↓
ГИП утверждает (2-я попытка)
        ↓
Основная: "В работе" ← разблокировка (может быть с задержкой)
```

Основная задача **не перезаписывает** данные:
- Все версии зависимой задачи видны в истории
- При утверждении ГИП явно указывает, какую версию использовать
- Инженер основной задачи видит пометку "версия N"

### Механизм подтверждения в основной задаче

После разблокировки основная задача требует явного подтверждения инженером:

```
Инженер видит в задаче:
"Данные от КЖ поступили (от [дата])"
        ↓
Проверяет актуальность
        ↓
Кликает "Данные приняты" (или добавляет комментарий)
        ↓
Только тогда может продолжить работу
        ↓
Статус остаётся "В работе" (но уже разблокирована)
```

Это важно, чтобы инженер **осознанно** начал работу с новыми данными.

## 6. ПРАВИЛА СИСТЕМЫ

### Неизменяемые правила

1. **Ответственный НИКОГДА НЕ ПУСТО**
   - У каждой задачи есть `assignee_id` (инженер)
   - ГИП не может оставить задачу без ответственного
   - RLS не позволит сохранить задачу без инженера

2. **Статус НЕ МЕНЯЕТСЯ БЕЗ РАЗРЕШЕНИЯ**
   - Инженер может менять статус только на определённые (в работе, на проверку)
   - Lead может менять только из "На проверке Lead'у"
   - ГИП может менять из "На проверке ГИПу" (приём/возврат)
   - Система может менять только автоматически (разблокировка)

3. **Возврат ВСЕГДА С ПРИЧИНОЙ**
   - Нельзя вернуть задачу статусом "На доработке" без замечания
   - Замечание должно иметь текст и приоритет
   - Инженер видит все причины перед тем, как снова отправить

4. **Задача МОЖЕТ БЫТЬ ЗАБЛОКИРОВАНА**
   - Статус "Ожидание данных" означает, что работа приостановлена
   - Это видно в Kanban и в списке задач
   - Инженер не может двигаться дальше, пока не будет разблокирована

5. **РАЗБЛОКИРОВКА ТОЛЬКО ЧЕРЕЗ ПОДТВЕРЖДЕНИЕ**
   - Автоматическая разблокировка: когда зависимая задача утверждена
   - Ручная разблокировка: ГИП может перевести в "В работе" и добавить комментарий
   - Инженер видит новые данные и должен явно подтвердить приём

## 7. ЛОГИКА БЛОКИРОВОК

### Когда задача блокируется

#### Вариант 1: Явная зависимость от другого отдела

```
ГИП создаёт задачу
        ↓
Понимает: нужны данные от КЖ
        ↓
Создаёт связь (task_dependencies)
        ↓
Статус задачи → "Ожидание данных"
        ↓
Инженер видит причину в описании задачи
        ↓
Не может начать работу до разблокировки
```

#### Вариант 2: Задача зависит от ревизии чертежа

```
Чертёж на проверке у ГИПа
        ↓
Задача расчётов, которая использует этот чертёж
        ↓
Статус → "Ожидание данных"
        ↓
После утверждения чертежа:
Система проверяет зависимости
        ↓
Статус → "В работе"
```

### Когда задача разблокируется

#### Автоматическая разблокировка

```
Зависимая задача утверждена ГИПом
        ↓
Система проверяет все "Ожидание данных" задачи
        ↓
Находит основную задачу с эту зависимостью
        ↓
Меняет статус с "Ожидание данных" → "В работе"
        ↓
Отправляет уведомление инженеру основной задачи
        ↓
Инженер видит в задаче: "Данные поступили [дата]"
```

**Запрос в БД:**
```sql
UPDATE tasks 
SET status = 'in_progress', 
    unblocked_at = NOW()
WHERE task_id = $parent_task_id
  AND status = 'awaiting_data'
  AND EXISTS (
    SELECT 1 FROM task_dependencies
    WHERE parent_id = $parent_task_id
      AND dependent_id = $dependent_task_id
      AND (SELECT status FROM tasks WHERE id = dependent_id) = 'approved'
  );
```

#### Ручная разблокировка

```
ГИП видит задачу в "Ожидание данных"
        ↓
Данные поступили через другой канал (по почте, вручную и т.д.)
        ↓
ГИП переводит задачу в "В работе"
        ↓
Добавляет комментарий: "Получены данные от [источник]"
        ↓
Инженер получает уведомление
        ↓
Может начать работу
```

### Контроль разблокировок

**ГИП видит:**
- Список всех заблокированных задач в проекте
- Причину блокировки (какая задача ждётся)
- Статус зависимой задачи (в работе, на проверке, утверждена)
- Сколько уже ждём (дней/часов)

**Lead видит:**
- Задачи в его отделе, от которых зависят другие задачи
- Что срок выполнения критичен для другого отдела
- Может попросить у ГИПа приоритет

**Инженер видит:**
- Почему его задача в "Ожидание данных"
- От какого отдела ждёт данные
- Когда были последние новости по зависимой задаче

## 8. ОТРАЖЕНИЕ В БД

### Таблица `tasks`

```sql
CREATE TABLE tasks (
  id BIGSERIAL PRIMARY KEY,
  project_id UUID NOT NULL REFERENCES projects(id),
  
  -- Ответственность
  assignee_id UUID NOT NULL REFERENCES app_users(id),
  assigned_by_gip_id UUID NOT NULL REFERENCES app_users(id),
  
  -- Статус (ENUM)
  status VARCHAR NOT NULL CHECK (status IN (
    'created',           -- 1. Создана
    'in_progress',       -- 2. В работе
    'review_lead',       -- 3. На проверке Lead
    'review_gip',        -- 4. На проверке ГИП
    'rework',            -- 5. На доработке
    'awaiting_data',     -- 6. Ожидание данных
    'approved'           -- 7. Утверждено
  )),
  
  -- Сроки
  deadline_at TIMESTAMP NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  approved_at TIMESTAMP,
  
  -- История проверок
  rework_count INT DEFAULT 0,
  returned_by_lead_at TIMESTAMP,
  returned_by_gip_at TIMESTAMP,
  
  -- Привязка к ревизии/документу
  drawing_id UUID REFERENCES drawings(id),
  revision_id BIGINT REFERENCES revisions(id),
  
  -- Описание
  title VARCHAR NOT NULL,
  description TEXT,
  
  -- Блокировка и зависимости
  dependency_resolved BOOLEAN DEFAULT FALSE,
  unblocked_at TIMESTAMP,
  
  -- RLS
  team_id UUID NOT NULL REFERENCES app_teams(id)
);
```

### Таблица `reviews` (замечания)

```sql
CREATE TABLE reviews (
  id BIGSERIAL PRIMARY KEY,
  task_id BIGINT NOT NULL REFERENCES tasks(id),
  
  -- Кто добавил замечание
  created_by_id UUID NOT NULL REFERENCES app_users(id),
  created_by_role VARCHAR NOT NULL CHECK (created_by_role IN ('lead', 'gip')),
  
  -- Критичность
  severity VARCHAR NOT NULL CHECK (severity IN ('minor', 'major', 'blocker')),
  
  -- Место в документе
  location VARCHAR,  -- например, "Лист 2, узел A3"
  
  -- Описание
  text TEXT NOT NULL,
  tag VARCHAR,  -- 'dimensioning', 'consistency', 'calculation', etc.
  
  -- Разрешение замечания
  resolved BOOLEAN DEFAULT FALSE,
  resolved_by_id UUID REFERENCES app_users(id),
  resolved_at TIMESTAMP,
  
  -- История
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);
```

### Таблица `task_dependencies` (связи между задачами)

```sql
CREATE TABLE task_dependencies (
  id BIGSERIAL PRIMARY KEY,
  
  -- Основная задача (ждёт данные)
  parent_task_id BIGINT NOT NULL REFERENCES tasks(id),
  
  -- Зависимая задача (поставляет данные)
  dependent_task_id BIGINT NOT NULL REFERENCES tasks(id),
  
  -- Описание
  required_data_description VARCHAR NOT NULL,
  
  -- Сроки
  deadline_at TIMESTAMP NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  resolved_at TIMESTAMP,
  
  -- RLS
  project_id UUID NOT NULL REFERENCES projects(id)
);
```

### RLS политики

```sql
-- Инженер видит только свои задачи
CREATE POLICY "engineer_own_tasks" ON tasks
  FOR SELECT USING (
    auth.uid() = assignee_id
    OR
    check_team_access(auth.uid(), team_id) IN ('gip', 'lead')
  );

-- Lead видит задачи своего отдела
CREATE POLICY "lead_team_tasks" ON tasks
  FOR SELECT USING (
    team_id = (SELECT team_id FROM app_users WHERE id = auth.uid())
  );

-- ГИП видит все задачи своего проекта
CREATE POLICY "gip_all_project_tasks" ON tasks
  FOR SELECT USING (
    project_id IN (
      SELECT project_id FROM app_users 
      WHERE id = auth.uid() AND role = 'gip'
    )
  );
```
