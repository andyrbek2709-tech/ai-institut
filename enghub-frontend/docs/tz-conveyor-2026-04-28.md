# ТЗ: Конвейер инженерного взаимодействия (EngHub)

> **Источник:** внешний эксперт, 28.04.2026.
> **Сохранено в репо как:** `enghub-main/docs/tz-conveyor-2026-04-28.md`

## 1. Цель

Превратить текущую систему (задачи, документы, замечания) в управляемый процесс (конвейер), где:
- понятно, где находится задача
- понятно, кто отвечает
- есть прозрачный цикл проверки и возврата
- есть учёт межотдельных зависимостей

## 2. Основная логика

### 2.1. Каждая задача имеет состояние (стадию)

Добавить отображаемые статусы:
- В работе (инженер выполняет)
- На проверке (нач. отдела)
- На проверке (ГИП)
- На доработке
- **Ожидание входных данных** ← новое
- Готово / Утверждено

### 2.2. У задачи всегда есть ответственный

| Стадия | Ответственный |
|---|---|
| В работе | инженер |
| На проверке (нач. отдела) | начальник отдела |
| На проверке (ГИП) | ГИП |
| На доработке | инженер |
| Ожидание входных данных | внешний отдел |

## 3. Действия пользователей (согласование)

### 3.1. Инженер
Может: отправить на проверку (нач. отдела)

### 3.2. Начальник отдела
Может:
- ✅ Принять → отправить ГИПу
- ❌ Вернуть → указать причину (обязательно)

### 3.3. ГИП
Может:
- ✅ Утвердить → задача завершена
- ❌ Вернуть → указать причину

### 3.4. Возврат задачи

При возврате:
- обязательно поле "причина"
- возможность: текст / прикрепить файл / скриншот

После возврата:
- статус = **На доработке**
- задача возвращается инженеру

## 4. Межотдельные зависимости (ключевой блок)

### 4.1. При создании задачи

Система задаёт вопрос: **"Нужны ли входные данные от других отделов?"**

Варианты:
- Нет → обычный процесс
- Да → открывается форма

### 4.2. Если выбрано "Да"

Пользователь указывает:
- от какого отдела
- что требуется
- срок

### 4.3. После этого

Система:
- Создаёт **связанную задачу** для другого отдела
- У текущей задачи статус: **Ожидание входных данных**

### 4.4. Когда данные получены

→ задача возвращается в **В работе**

## 5. Поведение интерфейса

### 5.1. В карточке задачи показывать
- текущий статус (крупно)
- кто сейчас отвечает
- история:
  - кто отправил
  - кто вернул
  - причины возврата

### 5.2. Кнопки должны быть контекстные

Инженер видит: "Отправить на проверку"

Нач. отдела видит: "Принять" / "Вернуть"

ГИП видит: "Утвердить" / "Вернуть"

👉 **другие кнопки не показывать**

## 6. Контроль ошибок (важно)

### 6.1. При создании задачи без зависимостей

Система спрашивает: **"Вы уверены, что не нужны данные от других отделов?"**

### 6.2. При возврате

Нельзя вернуть без причины.

## 7. Что НЕ делать
- не строить сложные схемы зависимостей
- не делать автоматические связи между отделами
- не добавлять сложные настройки

👉 всё через простой выбор пользователя

## 8. Как это ложится на текущую систему

Использовать существующее:
- задачи → как основа
- замечания → как возвраты
- документы → как результат
- трансмитталы → как выпуск

Добавляется только:
- статусы (стадии)
- логика переходов
- зависимости между задачами

## 9. Итоговое поведение системы

После внедрения:
- каждая задача проходит путь: выполнение → проверка → возврат / утверждение
- видно: где зависло, у кого зависло
- межотдельные связи фиксируются
- ГИП получает управляемый процесс
