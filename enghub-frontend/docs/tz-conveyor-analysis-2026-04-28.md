# Анализ ТЗ «Конвейер инженерного взаимодействия» vs текущий EngHub

> Подготовлено 28.04.2026, чтобы продолжить разговор после возвращения владельца.
> Источник ТЗ: `tz-conveyor-2026-04-28.md`

## 1. Что уже есть в EngHub (8 из ~12 пунктов покрыты)

✅ **Статусы задач** — почти всё что нужно:
- `todo` (≈ "В работе" — но семантика чуть иная: «ещё не взята»)
- `inprogress` ("В работе")
- `review_lead` ("На проверке (нач. отдела)")
- `review_gip` ("На проверке (ГИП)")
- `revision` ("На доработке")
- `done` ("Готово / Утверждено")

🟡 **Workflow переходы** в `taskWorkflowTransitions` (constants.ts) — есть, но без `awaiting_input`.

✅ **task_history** — триггер `task_changes_trigger` пишет смену статуса с `field_name`, `old_value`, `new_value`, `changed_by`. История в БД есть.

✅ **Контекстные кнопки по роли** — функция `getTaskActions` в App.tsx отдаёт actions по `task.status` + `role` пользователя. Логика правильная, но лишние кнопки могут показываться.

✅ **Возврат на доработку** — есть, но БЕЗ обязательного поля «причина». Сейчас можно вернуть пустым.

✅ **`assigned_to`** — поле для ответственного. По логике `assigned_to` + `status` = текущий ответственный.

✅ **RLS изоляция** — после фикса B4 каждый ГИП видит только свои проекты. Lead — свой отдел. Engineer — свои задачи.

## 2. Чего НЕТ совсем (нужно добавить)

🔴 **`awaiting_input`** — новый статус. Нужны:
- значение в БД (ENUM или text — у нас text, легко добавить)
- метка в `statusMap` в `constants.ts`
- цвет (предлагаю `#06b6d4` cyan — отличается от существующих)

🔴 **Межотдельные зависимости** — большая новая фича:
- Поле в задаче: «нужны входные данные?» (boolean)
- Связанная таблица `task_dependencies` или поле `parent_task_id`/`child_task_id`
- При создании задачи с зависимостью — автоматическое создание связанной задачи в другом отделе
- При выполнении связанной задачи → исходная переходит из `awaiting_input` → `inprogress`

🔴 **Прикрепление файла к причине возврата** — ни в одной форме нет file upload для возврата.

🔴 **Confirmation «Вы уверены что не нужны данные?»** — модалка при создании задачи без зависимостей.

🔴 **Полноценная история в карточке задачи (UI)** — данные есть в `listTaskHistory`, но в UI плохо отображается. Нужно: кто отправил, кто вернул, причины с timestamp.

🟡 **Обязательное поле «причина возврата»** — UI-валидация. Сейчас можно отправить пустым.

🟡 **Контекстные кнопки строго по роли** — `getTaskActions` отдаёт правильно, но иногда показываются лишние. Нужно проверить и зачистить.

## 4. Открытые вопросы для владельца

### Q1. Связанная задача — кому конкретно?

Когда инженер ВК говорит «нужны входные данные от КЖ» — система:
- (А) создаёт задачу-стаб без `assigned_to`, lead КЖ сам распределяет
- (Б) пытается автоматически найти инженера в КЖ с минимальной нагрузкой
- (В) сразу прикрепляет к lead КЖ, он распределяет

Какой вариант?

### Q2. Сроки связанной задачи

Если основная задача с дедлайном 30.05, а связанная нужна «к 15.05» — это **новое поле дедлайна для зависимости** или просто текстом «срок» в payload?

### Q3. Пересечение CONV-03 и T30a

CONV-03 (прикрепление файла к возврату) и T30a (загрузка ТЗ в проект) — **используют один Storage bucket или разные**?

Предлагаю один: `project-files/{project_id}/...` с подпапками `tz/`, `revisions/`, `comments/`. Согласен?

### Q4. Видимость связанной задачи

Если ВК создал задачу-зависимость для КЖ — это:
- (А) полноценная задача, КЖ работает с ней как с обычной
- (Б) **специальная** задача-«запрос данных» с урезанным набором действий

Я склоняюсь к (А) — проще, использует существующий workflow. Но (Б) даёт больше контроля.

### Q5. Что после получения данных от КЖ?

Когда КЖ выполнил связанную задачу → ВК-задача переходит из `awaiting_input` → `inprogress`. Но кто это инициирует?
- (А) КЖ нажимает «Готово» — система автоматически разблокирует ВК
- (Б) КЖ помечает «Передано в ВК», ВК сам подтверждает получение и нажимает «Возобновить работу»

(А) проще для пользователя, (Б) даёт ВК контроль.

## 5. Пересечения с уже принятыми решениями

- **DEC-AI-01** (AI-агенты) — независимо от этого ТЗ. Этот спринт можно делать параллельно.
- **T30a/T30b** (загрузка документов) — пересекается через CONV-03 (файл к возврату). Объединить Storage.
- **T30e** (передача файлов с задачами) — это **то же самое** что CONV-04+CONV-05. Объединить!
  - Файлы прикрепляются при создании зависимой задачи
  - Файлы возвращаются с готовым результатом
- **B4 RLS** (изоляция ГИПов) — должно работать с зависимостями. Если зависимость создаётся в **другой отдел того же проекта** — ОК. Между разными проектами зависимости не должно быть.

## 7. Что я понял и что согласен

- Идея «конвейер с явными стадиями и ответственными» — **отличная**. У нас сейчас почти всё есть, но без обязательной причины возврата и без межотдельных зависимостей.
- ТЗ ложится **очень аккуратно** на существующую модель — добавлять придётся не много, в основном расширять.
- Принцип «нет сложных схем, всё через простой выбор пользователя» — поддерживаю. CONV-04 будет одной формой с тремя полями, не визарт.
