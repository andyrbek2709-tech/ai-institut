# Сокращённый промт: быстрый прогон EngHub на одном проекте

> Скопируй всё ниже горизонтальной черты и вставь в облачный агент с доступом к браузеру (computer-use). Перед запуском подставь пароль админа в строку `__ВСТАВЬ__`.
>
> Это короткая версия — один проект, шесть тестовых пользователей, базовый цикл от создания до выпуска. Полный прогон с тремя проектами лежит в `enghub_test_prompt.md`.

---

## Роль

Ты — новый сотрудник проектного института, впервые открыл внутреннюю систему **EngHub**. Параллельно ты — UX-дизайнер и опытный инженер. Тебе нужно пройти один проект от начала до выпуска документа, посмотреть на программу двумя взглядами и записать отчёт.

## Цель

За один прогон:
1. Зайти под админом, потрогать админ-панель, завести команду из шести человек.
2. Пройти один проект (насосная станция КНС-3) от создания до выпущенного трансмиттала, с возвратом задачи на доработку и проверкой защит.
3. Записать отчёт по фазам и финальный итог.

## Доступ

- URL: `https://enghub-three.vercel.app`
- На старте есть **только одна** учётка — админ (`admin@enghub-test.ru` / `__ВСТАВЬ__`). Остальные пять пользователей ты заводишь сам.
- Единый стартовый пароль для созданных учёток: `Test2026!`

## Проект

**Насосная станция КНС-3 «Восток»**, шифр `КНС-3`, дедлайн +20 рабочих дней.
Отделы: ВК (главный), КМ.
Состав:
- задача `ВК-001` «Технологическая схема КНС, насосы»
- задача `КМ-001` «Расчёт фундамента под насосы»
По одному чертежу на задачу.

## Этап 0 — Админ заводит команду и проверяет возможности админ-панели

Это отдельный блок отчёта. Делается под T-0.1.

1. Логин под админом. Скриншот первого экрана.
2. Найди админ-панель. Скриншот.
3. Проверь список отделов. Должны быть `ВК` и `КМ`. Если нет — заведи. Запиши, что в админке вообще есть для отделов.
4. Заведи всех пятерых пользователей строго по таблице команды. ФИО — в формате `T-X.Y Фамилия И.О.`, email из таблицы, начальный пароль `Test2026!`.
5. **Возможности админ-панели — пройди по чек-листу, по каждому пункту скажи «есть / нет / работает / не работает»:**
   - создание пользователя: сколько кликов, какие обязательные поля, есть ли подсказки
   - валидация дубликата email при создании
   - смена пароля другому пользователю — есть ли отдельная кнопка
   - смена отдела у существующего пользователя
   - смена роли у существующего пользователя — пишется ли это куда-нибудь в журнал
   - возможность временно заблокировать пользователя без удаления
   - возможность удалить пользователя — что произойдёт с его задачами
   - управление отделами: создать, переименовать, удалить
   - есть ли журнал «кто и когда создал/изменил кого» (audit log)
   - есть ли список залогиненных сейчас пользователей или сессий
6. Стресс-проверки (записывай результат каждой):
   - создать пользователя с уже существующим email
   - создать руководителя без отдела
   - создать пользователя с пустой ролью
   - поменять пароль самому себе
   - удалить отдел, в котором есть сотрудники
7. Скриншоты: форма создания пользователя, итоговый список команды, любой ошибочный сценарий.
8. Оценка по 5 баллам: ясность / скорость / удобство админки.

После этого этапа все 6 учёток должны быть пригодны для логина.

## Фаза 1 — Логин под ГИПом и создание проекта

1. Выйди из админа, войди как T-1.1 (gip@enghub-test.ru / Test2026!).
2. Скриншот главного экрана ГИПа. Запиши: что нашёл за 30 секунд (проекты, задачи, AI-помощник, дашборд).
3. Создай проект КНС-3 по описанию выше.
4. Внутри проекта создай две задачи:
   - `ВК-001` — назначь руководителю ВК (T-2.5)
   - `КМ-001` — назначь руководителю КМ (T-2.1)
5. Скриншот канбана.

## Фаза 2 — Лид ВК распределяет задачу

1. Выйди, войди как T-2.5 Морозова.
2. Что она видит? Должна видеть `ВК-001`. Не должна видеть `КМ-001` (проверка защиты по отделу).
3. Открой `ВК-001`, нажми «Назначить инженеру», выбери T-3.5 Лебедеву.
4. Скриншот после назначения.

## Фаза 3 — Инженер ВК делает работу

1. Войди как T-3.5 Лебедева.
2. Открой `ВК-001`, жми «Взять в работу» — статус `inprogress`.
3. Перейди на «Чертежи», создай чертёж: код `ВК-001`, название «Технологическая схема КНС-3», ревизия `R0`. Любая тестовая ссылка на PDF.
4. Жми «Отправить на проверку» — статус `review_lead`.

## Фаза 4 — Возврат на доработку (обязательная проверка)

1. Войди как T-2.5 Морозова.
2. Открой `ВК-001`. На вкладке «Замечания» создай:
   - severity = critical: «Насосная группа не привязана к фундаменту КМ»
   - severity = major: «Не указан класс защиты IP»
3. Жми «На доработку» **без комментария** — программа должна отказать. Скриншот отказа.
4. Введи комментарий «См. два замечания, исправь и пришли R1». Жми снова — должно сработать.
5. Если первый шаг прошёл без отказа — это критичная находка, защита сломана.

## Фаза 5 — Исправление

1. Войди как T-3.5 Лебедева.
2. Прочитай комментарий и замечания.
3. На вкладке «Ревизии» — выпусти R1.
4. На вкладке «Замечания» закрой оба: open → in_progress → resolved.
5. Жми «Снова в работу», потом «Отправить на проверку».

## Фаза 6 — Приёмка и финал

1. Войди как T-2.5 Морозова, прими: «Утвердить → ГИПу».
2. Открой чертёж `ВК-001`, поставь статус `approved`.
3. Войди как T-1.1 Соколов (ГИП), завершить задачу: статус `done`.
4. Скриншот истории задачи — должен быть весь маршрут переходов.

## Фаза 7 — Параллельный цикл КМ (быстро, без возврата)

1. Под T-2.1 Петровой назначь `КМ-001` инженеру T-3.1 Орлову.
2. Под T-3.1 — взять в работу, создать чертёж `КМ-001`, отправить на проверку.
3. Под T-2.1 — утвердить → ГИПу.
4. Под T-1.1 — завершить, поставить чертёжу `approved`.

## Фаза 8 — Выпуск трансмиттала (обязательная проверка)

1. Под T-1.1 (ГИП) создай новый трансмиттал.
2. Добавь в состав оба чертежа (ВК-001 R1, КМ-001 R0).
3. **Тест блокировки.** Перед выпуском вернись в чертёж ВК-001 и снова открой одно critical-замечание (status → open). Иди в трансмиттал, жми «issued» — программа должна отказать с подсказкой «открыты critical». Скриншот отказа.
4. Закрой замечание. Снова жми «issued» — должно пройти.
5. Жми «PDF» — открывается окно печати. Проверь содержимое: шифр, список чертежей с ревизиями.
6. Если первый шаг прошёл без отказа — критичная находка.

## Фаза 9 — Стресс-сценарии безопасности (5 минут)

1. Под T-3.5 Лебедевой попробуй открыть админ-панель прямой ссылкой `/admin`. Должна отказать.
2. Под T-3.5 попробуй создать новый проект — кнопки быть не должно.
3. Под T-2.5 Морозовой (лид ВК) попробуй открыть `КМ-001` — задача чужого отдела, видеть её она не должна.
4. Скриншот любого нарушения.

## Формат блока отчёта по каждой фазе

```
### Фаза N — название

Что делал: 1–2 предложения.
Скриншоты: список с описаниями.
Что заметил как новичок: ...
Что заметил как специалист: ...
Оценка (1–5): ясность / скорость / удобство.
Найденные проблемы: ...
Что хорошо: ...
```

## Финальный отчёт

1. **Этап 0 — отчёт по админ-панели**: какие возможности есть/нет, что неудобно, чего критично не хватает.
2. **Общее впечатление** — три абзаца (новичок / специалист / итог).
3. **Сработали ли защиты:**
   - возврат без комментария блокируется? да / нет
   - выпуск трансмиттала с открытыми critical блокируется? да / нет
   - инженер видит чужие проекты или админ-панель? да / нет
4. **Критические находки** — что починить в первую очередь.
5. **Топ-5 предложений** по улучшению.
6. **Профили ролей** — короткий итог по каждой из четырёх ролей (админ / ГИП / лид / инженер).

## Дисциплина

- Один скриншот = одно действие.
- Время на фазу — не больше 10 минут.
- При поломке — записать и идти дальше, не пытаться чинить.
- При падении программы — приложить текст ошибки и URL.

## Критерии «прошёл / не прошёл»

✓ Прошёл, если:
- команда заведена под админом
- ВК-001 прошла полный цикл с возвратом и попала в done
- защита возврата без комментария сработала
- защита выпуска при открытых critical сработала
- T-3.5 не увидела чужие проекты и не смогла открыть админку

✗ Не прошёл, если хоть одна из защит выше промолчала, или цикл не довёлся до выпущенного трансмиттала.
