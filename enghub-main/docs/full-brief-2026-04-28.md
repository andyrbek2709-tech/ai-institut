## 1. Что такое EngHub в одном абзаце

EngHub — это **внутренняя цифровая платформа для проектных институтов**: связывает приёмку техзадания, постановку задач по отделам, выпуск чертежей с ревизиями, согласование замечаний, формирование трансмитталов, видеосовещания, табель, AI-помощника и нормативную базу с RAG-поиском. Заменяет связку «Excel + Bitrix + Outlook + Skype». Пользуются 4 роли: Администратор, ГИП (Главный инженер проекта), Руководитель отдела, Инженер.

## 4. Workflow задачи (главный цикл)

```
todo  ──[engineer берёт]──> inprogress  ──[engineer завершил]──> review_lead
                                                                       │
                                                                       │ (lead одобрил)
                                                                       ▼
                                                                 review_gip
                                                                       │
                                              (ГИП одобрил)            │ (ГИП вернул)
                                                       ▼               ▼
                                                     done          revision
                                                                       │
                                                                       │ (engineer переделал)
                                                                       ▼
                                                                 inprogress
```

- На любом этапе lead/gip могут вернуть на `revision` с комментарием.
- Перевод в `done` доступен **только ГИПу проекта**.
- При каждом переходе в `task_history` пишется запись (старый/новый статус, кто менял).
- Триггер `task_changes_trigger` логирует UPDATE автоматически.

## 5. Workflow чертёж → ревизия → замечание → трансмиттал

```
1. Создание чертежа
   drawings: { name, code, status='draft' }
   ↓
2. Ревизия (R0, R1, R2…)
   revisions: { drawing_id, code='R0', author_id, comment }
   ↓
3. Замечание (от другого инженера/lead/gip)
   reviews: { drawing_id, project_id, severity=critical|major|minor, status='open' }
   ↓
4. Обсуждение в треде
   review_comments: { review_id, author_id, message }
   ↓
5. Закрытие замечания (автор согласился / отклонил)
   reviews.status = 'resolved' | 'rejected'
   ↓
6. Новая ревизия чертежа (R1)
   revisions: { drawing_id, code='R1', parent_revision_id }
   ↓
7. Формирование трансмиттала (комплект чертежей для выпуска)
   transmittals: { project_id, code='TR-ХХХ-001', status='draft' }
   transmittal_items: { transmittal_id, drawing_id, revision_id }
   ↓
8. Выпуск трансмиттала
   transmittals.status = 'issued'
   (toast: «Статус трансмиттала изменён: Выпущен»)
   ↓
9. Доставка (опционально)
   transmittals.status = 'delivered' | 'cancelled'
```

**Автонумерация:** `TR-{project_code}-{NNN}` — счётчик глобальный по проекту.

## 6. AI Copilot — что умеет

**Endpoint:** `POST /api/orchestrator`
**Модели:** GPT-4o-mini (декомпозиция, NL-search), Claude (анализ, генерация задач)

### Поддерживаемые intent'ы (роль-зависимо)

| Intent | gip | lead | engineer |
|---|:---:|:---:|:---:|
| `create_tasks` (декомпозиция проекта на задачи) | ✅ | ✅ | ✅ |
| `create_drawing` / `update_drawing` | ✅ | ✅ | ❌ |
| `drawing_revision` (новая ревизия) | ✅ | ✅ | ❌ |
| `create_review` (создать замечание) | ✅ | ✅ | ✅ |
| `create_transmittal` | ✅ | ✅ | ❌ |
| `workflow_transition` (валидация перехода статуса) | ✅ | ✅ | ✅ |
| `project_insights` (анализ рисков) | ✅ | ✅ | ✅ |
| `smart_decompose` (умное разбиение задачи) | ✅ | ✅ | ❌ |
| `compliance_check` (проверка нормативки) | ✅ | ✅ | ✅ |
| `generate_report` (отчёт для заказчика) | ✅ | ✅ | ❌ |
| `nl_search` (поиск на русском) | ✅ | ✅ | ✅ |
| `risk_forecast` | ✅ | ✅ | ❌ |

### Защита AI-апплая

Все опасные действия (создание/изменение задач, чертежей, ревизий) идут **через approval-гейт**: AI готовит payload в `ai_actions` со `status='pending'`, пользователь нажимает «Утвердить» — только тогда применяется.

`validateApplyAction.ts` дополнительно проверяет:
- AI **не может** назначать конкретного инженера (поле `assigned_to` обязано быть пустым).
- Длина названия задачи — до 200 символов.
- Дата дедлайна валидна.
- Роль engineer не может вызывать opcoдe `create_drawing`, `create_transmittal` и т.д.

### RAG (нормативка)

- **Хранилище:** таблица `normative_chunks` с pgvector embeddings от OpenAI
- **Запрос:** `action='search_normative'` → RPC `search_normative(query_embedding, match_count)`
- **Источники:** ГОСТы, СП, СНиП, ВНТП — загружены в `normative_docs`, индексируются автоматом

## 7. Видеосовещания (LiveKit)

**Endpoint:** `POST /api/meeting-token`

Сценарий:
1. Пользователь открывает таб «Совещание» в проекте.
2. Фронт SELECT'ит `video_meetings WHERE project_id=X AND ended_at IS NULL` — если есть активная, подключает.
3. Если нет — INSERT новой `video_meetings` (RLS пропускает: участник проекта).
4. Запрос `/api/meeting-token` с meeting_id → сервер выдаёт LiveKit JWT.
5. Фронт подключается к `wss://institutepro-m9n7ay5f.livekit.cloud` с JWT.
6. В `video_meeting_participants` upsert'ится строка для текущего пользователя.
7. Чат сохраняется в `video_meeting_chat_messages` с RLS «только участники видят».

**Важно (исправлено 28.04):** RLS не зацикливается между `video_meetings ↔ video_meeting_participants` — введён SECURITY DEFINER хелпер `is_meeting_participant(uuid)`.

## 8. Окружающие модули

- **Совещания (протоколы)** — отдельно от LiveKit. Таблица `meetings` хранит планы и протоколы. PDF-экспорт через `api/spec-export`.
- **Табель (timelog)** — `time_entries` с user_id, task_id, дата, часы. Дашборд загрузки отделов считается отсюда.
- **Дашборд ГИПа** — Risk-меттрики, S-кривая прогресса проектов, RACI-матрица. Только при `role='gip'`.
- **Telegram-бот** — `api/telegram.js`. Уведомления о новых задачах, замечаниях, статусах.
- **Инженерные расчёты** — каталог из 90 формул по разделам ТХ/ЭО/ВК/ПБ/ГП. Экспорт результатов в `.docx`.
- **Спецификации** — таблица оборудования с привязкой к проекту.
- **Нормативка (RAG)** — поиск по ГОСТам/СП на естественном языке.

## 12. День из жизни команды (полный сценарий)

> Ниже — реальный сценарий, который можно прогнать как QA-тест. Все шаги работают на проде сейчас.

### Утро ГИПа

1. ГИП логинится как `gip1@enghub-test.ru / TestGip1!`.
2. На дашборде видит свои 0 проектов (новый ГИП). Нажимает «+ Новый проект».
3. Заполняет: «ТЭЦ-7 Реконструкция», код «ТЭЦ7-2026-01», отделы: КМ, ОВ, ЭС, дедлайн 31.12.2026.
4. Проект создан — автоматически прикреплён к ГИПу как `gip_id=59`.
5. ГИП открывает карточку проекта, видит вкладки «Совещание / Задачи / Чертежи / …».
6. Открывает «AI Copilot» → пишет «Разработай план задач для реконструкции ТЭЦ».
7. Copilot готовит 8 задач (по 2 на каждый отдел) → ГИП утверждает.
8. Задачи появляются в табе «Задачи» со статусом `todo`.

### День руководителя отдела

9. Lead КМ заходит в систему, видит проект «ТЭЦ-7» в своём списке (потому что есть задачи его отдела).
10. Видит 2 задачи отдела КМ. Назначает одну инженеру А, вторую инженеру Б.

### Работа инженера

11. Инженер А заходит — видит 1 свою задачу со статусом `todo`.
12. Открывает её, читает описание. Нажимает «▶ Взять в работу» → `inprogress`.
13. Инженер создаёт чертёж «АР-101» во вкладке «Чертежи».
14. Создаёт ревизию `R0` для этого чертежа.
15. По окончании работы нажимает «↑ Отправить на проверку» → статус `review_lead`.

### Замечание от смежника

16. Инженер из отдела ОВ открывает чертежи → видит чертёж АР-101.
17. Создаёт замечание: «По узлу А-3 нет привязки к ОВ-сети, severity=major».
18. Инженер А получает уведомление в Telegram (если настроен).

### Возврат на доработку

19. Lead КМ открывает задачу в `review_lead`, видит замечание.
20. Нажимает «✗ На доработку» с комментарием → статус `revision`.
21. Инженер А получает уведомление, переделывает.
22. Создаёт ревизию `R1`. Снова отправляет на проверку.

### Согласование и выпуск

23. Lead одобряет → `review_gip`.
24. ГИП открывает задачу, проверяет, нажимает «✓ Завершить задачу» → `done`.
25. ГИП формирует трансмиттал «TR-ТЭЦ7-2026-01-001», добавляет туда чертёж АР-101 ревизия `R1`.
26. ГИП меняет статус трансмиттала на «Выпущен» → toast «Статус трансмиттала изменён: Выпущен».
27. Документ ушёл заказчику.

### Совещание по итогам

28. ГИП открывает таб «Совещание», нажимает «Войти на встречу».
29. LiveKit подключает камеру/микрофон, в комнату заходят Lead и инженеры.
30. Обсуждают, ГИП пишет в чате замечания, при выходе из комнаты автоматически создаётся протокол.

### Конец дня

31. Все вносят часы в табель.
32. Каждое утро в Telegram приходит digest: «Вчера закрыто N задач, выпущен 1 чертёж, открыто 1 замечание».
