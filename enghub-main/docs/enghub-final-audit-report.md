## 📋 Обнаруженные проблемы

### 🔴 КРИТИЧЕСКИЕ (3)

#### 1. Нарушение изоляции проектов (RLS)
- **Категория:** RLS (Row Level Security)
- **Проблема:** Все роли (gip, lead, engineer, lead_engineer, observer) видят проекты всех ГИПов
- **Детали:**
  - ГИП1 (ID: 4) создал 3 проекта
  - ГИП2 (ID: 9) создал 1 проект
  - Все роли видят 4 проекта (и те и другие)
- **Должно быть:**
  - ГИП1 видит только свои 3 проекта
  - ГИП2 видит только свой 1 проект
  - Начальник видит только назначенные проекты
  - Инженер видит только назначенные проекты

#### 2. RLS политики не настроены для projects
- **Категория:** RLS
- **Проблема:** RLS для таблицы `projects` не фильтрует по назначениям
- **Детали:** Нет политики, которая проверяла бы `project_assignments`
- **Должно быть:**
  ```sql
  CREATE POLICY "Users can view assigned projects"
  ON projects FOR SELECT
  USING (
    gip_id = auth.uid()::int OR
    id IN (SELECT project_id FROM project_assignments
           WHERE user_id = auth.uid()::int AND is_active = TRUE)
  );
  ```

#### 3. Таблица project_assignments пуста
- **Категория:** Database / Workflow
- **Проблема:** Назначения на проекты не созданы
- **Детали:** В таблице `project_assignments` нет записей
- **Должно быть:**
  - ГИПы должны быть назначены как GIP на свои проекты
  - Начальники отделов должны быть назначены на проекты
  - Инженеры должны быть назначены на проекты

### 🟡 СРЕДНИЕ (1)

#### 1. ГИП не может создать проект
- **Категория:** Workflow
- **Проблема:** Ошибка при создании проекта - deadline constraint
- **Детали:** `null value in column "deadline" violates not-null constraint`
- **Причина:** Скрипт не передает deadline или передает пустое значение
- **Должно быть:**
  - Поле deadline может быть nullable или должно передаваться значение
  - Альтернативно: API должен устанавливать deadline по умолчанию

## 🔧 Необходимые исправления

### Шаг 1: Исправить RLS политики для projects

Выполните в Supabase Dashboard → SQL Editor:

```sql
-- Удаляем старые политики
DROP POLICY IF EXISTS "Users can view their own or assigned projects" ON projects;

-- Создаем правильную политику
CREATE POLICY "Users can view their own or assigned projects"
ON projects FOR SELECT
USING (
  -- Создатель проекта (ГИП)
  gip_id = (SELECT id FROM app_users WHERE supabase_uid = auth.uid()) OR
  -- Назначен на проект через project_assignments
  id IN (
    SELECT project_id
    FROM project_assignments pa
    JOIN app_users au ON pa.user_id = au.id
    WHERE au.supabase_uid = auth.uid() AND pa.is_active = TRUE
  ) OR
  -- Администратор
  (SELECT role FROM app_users WHERE supabase_uid = auth.uid()) = 'admin'
);
```

### Шаг 2: Создать назначения для существующих проектов

```sql
-- Назначаем ГИПов на свои проекты
INSERT INTO project_assignments (project_id, user_id, role, assigned_by, is_active)
SELECT
  p.id as project_id,
  au.id as user_id,
  'gip' as role,
  au.id as assigned_by,
  TRUE as is_active
FROM projects p
JOIN app_users au ON au.supabase_uid = p.gip_id::text
WHERE NOT EXISTS (
  SELECT 1
  FROM project_assignments pa
  WHERE pa.project_id = p.id
    AND pa.user_id = au.id
    AND pa.role = 'gip'
    AND pa.is_active = TRUE
);

-- Если у проекта есть departments в массиве depts, назначаем начальников
INSERT INTO project_assignments (project_id, user_id, role, assigned_by, is_active)
SELECT DISTINCT
  p.id as project_id,
  d.head_id as user_id,
  'lead' as role,
  (SELECT au.id FROM app_users au WHERE au.supabase_uid = p.gip_id::text) as assigned_by,
  TRUE as is_active
FROM projects p
CROSS JOIN LATERAL unnest(p.depts) AS dept_id
JOIN departments d ON d.id = dept_id::int
WHERE d.head_id IS NOT NULL
  AND NOT EXISTS (
    SELECT 1
    FROM project_assignments pa
    WHERE pa.project_id = p.id
      AND pa.user_id = d.head_id
  );
```

### Шаг 3: Сделать поле deadline nullable (опционально)

```sql

ALTER TABLE projects ALTER COLUMN deadline DROP NOT NULL;
```

ИЛИ:
