## Что починено в этой сессии

### Bug #2 — Modal Stage 4b скрыт за Task Detail (z-index)
- **Симптом:** кнопка «🔗 Запросить данные» нажимается, в DOM появляется второй overlay с титулом «Запрос данных у смежного отдела», но визуально его не видно.
- **Причина:** оба `Modal` используют CSS-класс `.modal-overlay` с `z-index: 1000`. Модал Stage 4b в DOM рендерится **раньше** Task Detail модала (App.tsx:1548 vs 1568), и при равном z-index выигрывает порядок DOM — Task Detail рисуется поверх.
- **Фикс:**
  - `src/components/ui.tsx` — добавлен опциональный пропс `Modal.topmost`. Если `true` — overlay получает inline `style={{ zIndex: 1100 }}`.
  - `src/App.tsx` — `<Modal title="Запрос данных у смежного отдела" ... topmost>`.
- **Статус:** локально исправлено, **ожидает push** (bash в sandbox временно даёт «no space left on device»).

## Известные проблемы (требуют отдельной работы)

### 🚨 Bug #3 — CRITICAL: service_role ключ Supabase утекает в клиент
- **Подтверждение:** скачал `https://enghub-three.vercel.app/static/js/main.5d8fd3a7.js` (2.19 MB) и распарсил все JWT — найдено **3 токена с `role: "service_role"`** + один новый формат `sb_secret_*`.
- **Что это даёт атакующему:** полный bypass Row Level Security. Любой посетитель открывает DevTools → Network → main JS → копирует токен → делает `Authorization: Bearer <service_key>` к Supabase REST/Storage и читает/пишет/удаляет любые данные минуя политики.
- **Источник в коде:**
  - `src/api/supabase.ts:5` — `const SERVICE_KEY = process.env.REACT_APP_SUPABASE_SERVICE_KEY || ''` (любая `REACT_APP_*` переменная попадает в bundle).
  - `src/api/supabaseClient.ts:5` — то же.
  - `src/components/ActivityFeed.tsx` — импортирует и использует `SERVICE_KEY` напрямую через fetch к `/rest/v1/activity_log`.
  - `src/api/supabase.ts` — функции `signProjectFileUrl`, `removeFromBucket`, `createNotification` ходят с `Bearer SERVICE_KEY` прямо из браузера.
- **Что сделать (план миграции):**
  1. В Vercel Dashboard → Project Settings → Environment Variables: **удалить** `REACT_APP_SUPABASE_SERVICE_KEY`.
  2. Перенести admin-операции в существующие серверные функции (`api/*.js`), они уже используют `process.env.SUPABASE_SERVICE_KEY` (без `REACT_APP_` префикса) безопасно: `api/_spec_helpers.js`, `api/orchestrator.js`, `api/telegram.js`, `api/weekly-digest.js`, `api/meeting-token.js`.
  3. Создать новые endpoints: `/api/activity-log` (читает activity_log, проверяет JWT пользователя), `/api/sign-file-url` (подписанные URL), `/api/notifications/create`.
  4. На клиенте (`ActivityFeed.tsx`, `supabase.ts`) переключиться на anon-key + user JWT и эти proxy-endpoints.
  5. **Сразу после деплоя ротировать service_role key** в Supabase Dashboard (старый утёк).
- **Не правил в этой сессии** — частичный фикс ломает функционал; нужен ~день работы и тестирования.

### ⚠ Bug #1 — Race condition: KPI инженера показывают 0 при первой загрузке
- **Симптом:** сразу после логина «Мои задачи / В работе / На проверке / Завершено» показывают 0/0/0/0, при этом виджет EngineerDashboard правильно показывает «Мои задачи (1)».
- **Причина:** `loadAllTasks(pid)` (App.tsx:424) фильтрует `tasks` по `currentUserData?.id`. Если профиль пользователя ещё не пришёл — `myId === ''`, фильтр отбрасывает всё, и `tasks` остаётся пустым.
- **Текущая защита:** есть `useEffect` с deps `[currentUserData?.id]` (App.tsx:376), который перезагружает задачи после получения профиля. На практике race разрешается через ~1 сек.
- **Минор**, фикс не делаю — но стоит пометить как UX-сглаживание (показать skeleton в KPI пока currentUserData=null).

### ⚠ Bug #4 — Lead/Engineer Dashboard ограничены активным проектом
- **Симптом:** Pravdukhin (Lead отдела ЭС) видит «Нагрузка инженеров (3) — все 0», хотя у Трошина есть открытая задача в проекте «QA Тест 29.04». Причина — активный проект в момент просмотра был не QA Тест, а старый «1».
- **Причина:** `allTasks` грузится через `loadAllTasks(activeProject.id)` — это задачи **только одного активного проекта**. Виджеты dashboard'ов получают этот же массив.
- **Архитектурное:** для real Lead/Engineer dashboard нужен отдельный fetch всех задач, где `assigned_to` входит в множество инженеров отдела (Lead) или равен currentUser.id (Engineer), без привязки к одному проекту.
- **Не фикшу** — переработка scope.
