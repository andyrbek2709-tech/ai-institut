## 1. РОЛЬ ОРКЕСТРАТОРА

### Что делает оркестратор

**System Orchestrator** — это фоновый механизм, который **управляет системой автоматически**, без участия людей.

**Основные функции:**

```
ОРКЕСТРАТОР СЛЕДИТ ЗА:

1. Статусами всех задач в реальном времени
2. Блокировками (зависимостями между задачами)
3. Дедлайнами (сроками выполнения)
4. Временем в статусах (сколько уже на проверке)
5. Изменениями (когда что-то произошло)
6. Условиями для переходов (можно ли перейти в другой статус)

ОРКЕСТРАТОР ИНИЦИИРУЕТ:

→ Переходы статусов (автоматические)
→ Уведомления (к людям в систему и Telegram)
→ Алерты (если что-то идёт не так)
→ Эскалации (если Lead не проверил 24 часа)
→ Разблокировки (когда зависимая задача готова)
```

### Архитектурный паттерн

```
┌─────────────────────────────────────┐
│  СОБЫТИЯ                            │
│  (что произошло в системе)          │
└──────────────┬──────────────────────┘
               │
               ↓
┌─────────────────────────────────────┐
│  ОРКЕСТРАТОР                        │
│  • слушает события                  │
│  • вычисляет новые статусы          │
│  • проверяет правила                │
│  • принимает решения                │
└──────────────┬──────────────────────┘
               │
               ↓
┌─────────────────────────────────────┐
│  ДЕЙСТВИЯ                           │
│  • изменить статус                  │
│  • отправить уведомление            │
│  • создать запись в истории         │
└─────────────────────────────────────┘
```

## 2. СОБЫТИЯ СИСТЕМЫ (EVENTS)

### 2.1 События от пользователей

| Событие | Кто инициирует | Данные | Последствие |
|---|---|---|---|
| **task_created** | ГИП | task_id, project_id, assignee_id, deadline | Задача создана, статус → "Создана" |
| **task_started** | Инженер | task_id, timestamp | Статус → "В работе" |
| **file_attached** | Инженер | task_id, file_id | Файл добавлен к задаче |
| **submitted_for_review** | Инженер | task_id, timestamp | Статус → "На проверке Lead" |
| **review_opened** | Lead / ГИП | task_id, reviewer_id | Кто-то открыл карточку для проверки |
| **review_comment_added** | Lead / ГИП | task_id, comment_id, severity, text | Добавлено замечание |
| **review_resolved** | Инженер / Lead | comment_id | Замечание отмечено как разрешённое |
| **task_accepted_by_lead** | Lead | task_id, timestamp | Статус → "На проверке ГИПу" |
| **task_returned_by_lead** | Lead | task_id, reason | Статус → "На доработке" |
| **task_approved_by_gip** | ГИП | task_id, timestamp | Статус → "Утверждено" |
| **task_returned_by_gip** | ГИП | task_id, reason | Статус → "На доработке" |
| **dependency_created** | ГИП | parent_task_id, dependent_task_id | Основная: "Ожидание данных" |
| **dependent_task_approved** | Система | dependent_task_id | → Разблокировать основную |

### 2.2 События от системы (Auto-generated)

| Событие | Когда возникает | Действие |
|---|---|---|
| **deadline_approaching_2d** | Дедлайн - 2 дня | Отправить напоминание Lead / инженеру |
| **deadline_approaching_1d** | Дедлайн - 24 часа | Отправить СРОЧНОЕ напоминание |
| **deadline_exceeded** | Дедлайн прошёл | Красный флаг, уведомление ГИПа |
| **blocking_24h** | Блокировка > 24 часа | Уведомление Lead заблокированной задачи |
| **blocking_48h** | Блокировка > 48 часов | Уведомление ГИПа + Lead |
| **blocking_72h** | Блокировка > 72 часа | 🚨 КРИТИЧЕСКИЙ ALERT ГИПу |
| **review_timeout_lead_24h** | На проверке Lead > 24ч | Напоминание Lead |
| **review_timeout_lead_48h** | На проверке Lead > 48ч | Alert ГИПу |
| **review_timeout_gip_24h** | На проверке ГИП > 24ч | Alert ГИПу |
| **rework_cycle_3+** | 3+ попытки | Оповещение ГИПу (проблема с инженером?) |
| **heartbeat_check** | Каждый час (фоновая проверка) | Проверить все условия эскалации |

## 3. ТРИГГЕРЫ И ПЕРЕХОДЫ

### 3.1 Триггер: Создание задачи

```
СОБЫТИЕ: task_created
────────────────────────────────────

ПРОВЕРКИ:
1. assignee_id НЕ NULL? ✓ (обязательно)
2. deadline_at > NOW()? ✓ (дедлайн в будущем)
3. project_id существует? ✓

ЕСЛИ все ОК:
→ Статус задачи = "created"
→ Записать в БД: created_at, assigned_by, assignee
→ Отправить уведомление инженеру: "Вам назначена новая задача"

ЕСЛИ есть зависимость (dependency указана):
→ Статус = "awaiting_data" ← автоматически!
→ Создать зависимую задачу (в другом отделе)
→ Отправить уведомление Lead затронутого отдела

ОТПРАВИТЬ УВЕДОМЛЕНИЕ:
- Инженер (Telegram): "Новая задача [NAME], дедлайн [DATE]"
- IN-APP: Уведомление появится в центре уведомлений
```

### 3.2 Триггер: Инженер начинает работу

```
СОБЫТИЕ: task_started
────────────────────────────────────

ПРОВЕРКИ:
1. Статус = "created" или "awaiting_data"? ✓
2. Если "awaiting_data": зависимость ещё не разрешена ✗
   → Отправить ошибку: "Задача заблокирована, дождитесь данных"

ЕСЛИ ОК (статус = "created"):
→ Статус = "in_progress"
→ Записать: started_at = NOW()
→ История: [инженер] начал работу

ОТПРАВИТЬ УВЕДОМЛЕНИЕ:
- ГИП (IN-APP): "Инженер [NAME] начал работать над [TASK]"
- Lead (опционально): видит в своей очереди
```

### 3.3 Триггер: Инженер отправляет на проверку

```
СОБЫТИЕ: submitted_for_review
────────────────────────────────────

ПРОВЕРКИ:
1. Статус = "in_progress" или "rework"? ✓
2. Есть хотя бы один файл прикреплён? ✓
   Если нет: ошибка "Прикрепите файл перед отправкой"

ЕСЛИ ОК:
→ Статус = "review_lead"
→ Записать: submitted_to_lead_at = NOW()
→ Счётчик попыток: +1 (если это повторная отправка)
→ История: [инженер] отправил на проверку Lead

ОТПРАВИТЬ УВЕДОМЛЕНИЕ:
- Lead (Telegram + IN-APP): 
  "📦 Новая задача на проверку: [TASK] (инженер: [NAME])"
  Со ссылкой на очередь Lead
  
- ГИП (IN-APP): видит в статистике
```

### 3.4 Триггер: Lead добавляет замечание

```
СОБЫТИЕ: review_comment_added
────────────────────────────────────

ДЕЙСТВИЕ:
→ Сохранить замечание в БД (reviews таблица)
→ Статус замечания = "unresolved"
→ Если severity = "blocker": статус задачи ОСТАЁТСЯ "review_lead"
  (невозможно отправить ГИПу с blocking замечанием)

ОТПРАВИТЬ УВЕДОМЛЕНИЕ:
- Инженер (IN-APP): 
  "💬 Новое замечание от Lead: [TEXT]"
  Ссылка на замечание
  
- ГИП (IN-APP): видит замечание (если уже на его проверке)
```

### 3.5 Триггер: Lead принимает результат

```
СОБЫТИЕ: task_accepted_by_lead
────────────────────────────────────

ПРОВЕРКИ:
1. Статус = "review_lead"? ✓
2. Все замечания severity = "blocker" разрешены? ✓
   Если есть unresolved blocker: ошибка
   "Не можно отправить с blocking замечаниями"

ЕСЛИ ОК:
→ Статус = "review_gip"
→ Записать: submitted_to_gip_at = NOW()
→ История: [Lead] одобрил, отправил ГИПу

ОТПРАВИТЬ УВЕДОМЛЕНИЕ:
- ГИП (Telegram + IN-APP):
  "✅ Готово к утверждению: [TASK]"
  "Качество проверено Lead [NAME]"
  Со ссылкой на очередь ГИПа
  
- Инженер (IN-APP): "Ваша работа одобрена Lead, ждёт ГИПа"
```

### 3.6 Триггер: Lead / ГИП возвращают на доработку

```
СОБЫТИЕ: task_returned
────────────────────────────────────

ПРОВЕРКИ:
1. Статус = "review_lead" или "review_gip"? ✓
2. Есть хотя бы одно неразрешённое замечание? ✓
   Если нет: ошибка "Добавьте замечание перед возвратом"

ЕСЛИ ОК:
→ Статус = "rework"
→ Записать: returned_at = NOW()
→ returned_by = "lead" / "gip"
→ Счётчик возвратов: +1
→ Если 3+ возврата: ГИП получает alert "Проблема с инженером?"
→ История: [Lead/ГИП] вернул, причина [LIST OF COMMENTS]

ОТПРАВИТЬ УВЕДОМЛЕНИЕ:
- Инженер (Telegram + IN-APP):
  "❌ Задача возвращена на доработку"
  "Замечания: [COUNT]"
  Ссылка на карточку
  
- Lead (если вернул ГИП): "ГИП вернул задачу от [инженера]"
```

### 3.7 Триггер: ГИП утверждает результат

```
СОБЫТИЕ: task_approved_by_gip
────────────────────────────────────

ПРОВЕРКИ:
1. Статус = "review_gip"? ✓
2. Все blocker замечания разрешены? ✓

ЕСЛИ ОК:
→ Статус = "approved"
→ Записать: approved_at = NOW()
→ approved_by_gip_id = [гип]
→ История: [ГИП] утвердил, дата и подпись

АВТОМАТИЧЕСКИЕ ДЕЙСТВИЯ:
→ РАЗБЛОКИРОВАТЬ все зависимые задачи:
  Если есть task_dependencies где dependent_id = этой задаче
  → Для каждой parent_task_id:
    IF parent.status = "awaiting_data"
    → Изменить на "in_progress"
    → Записать: unblocked_at = NOW(), dependency_resolved = TRUE
    → Отправить уведомление инженеру основной задачи

ОТПРАВИТЬ УВЕДОМЛЕНИЕ:
- Инженер (Telegram + IN-APP):
  "✅ Ваша задача [TASK] утверждена!"
  
- Lead (IN-APP): "Задача [TASK] утверждена"

- ГИП: видит в Dashboard (добавить в статистику "Утверждено")

- Инженер ЗАВИСИМОЙ задачи (если разблокирована):
  "🎉 Данные готовы: [зависимая задача] утверждена!"
  "Вы можете продолжить работу над [основная задача]"
```

### 3.8 Триггер: Разблокировка зависимой задачи

```
СОБЫТИЕ: dependent_task_approved
────────────────────────────────────

ЭТО АВТОМАТИЧЕСКОЕ ДЕЙСТВИЕ В ОРКЕСТРАТОРЕ

ЛОГИКА:
1. Когда задача переходит в статус "approved"
2. Проверить: есть ли task_dependencies где dependent_id = этой задаче
3. Для каждого parent_id:

   IF parent.status = "awaiting_data"
      AND зависимость.resolved_at IS NULL
   THEN:
      → Обновить parent.status = "in_progress"
      → Записать: unblocked_at = NOW()
      → Записать: task_dependencies.resolved_at = NOW()

ОТПРАВИТЬ УВЕДОМЛЕНИЕ:
- Инженер основной задачи (Telegram + IN-APP):
  "🎉 Блокировка снята!"
  "Данные от [отдел] готовы: [название зависимой]"
  "Вы можете продолжить работу"
  
- ГИП (Dashboard): блокировка исчезает из списка

- Lead (если это его инженер): видит задачу разблокированной
```

## 4. КОНТРОЛЬ БЛОКИРОВОК (BLOCKING LOGIC)

### 4.1 Когда задача блокируется

```
УСЛОВИЕ 1: При создании задачи, если указана зависимость
─────────────────────────────────────────────────────────

ГИП создаёт задачу и указывает:
"Эта задача требует данные от [другой задачи]"

ОРКЕСТРАТОР:
→ Статус задачи = "awaiting_data"
→ Записать в task_dependencies:
   parent_task_id = эта задача
   dependent_task_id = [другая задача]
   deadline = [когда нужны данные]
   
ИНЖЕНЕР ВИДИТ:
"Задача заблокирована. Ждёт: [другая задача] (дедлайн: [дата])"
"Вы сможете начать, когда данные будут готовы"
```

### 4.2 Статус блокировки в реальном времени

```
ОРКЕСТРАТОР ОТСЛЕЖИВАЕТ:

blocked_since = WHEN(status CHANGED TO 'awaiting_data')
blocked_duration = NOW() - blocked_since

И ПОКАЗЫВАЕТ:
- На карточке: "Блокирована с [время] ([N] часов/дней)"
- На канбане: красный индикатор 🔴
- В Dashboard: список всех заблокированных задач с временем
```

### 4.3 Контроль времени блокировки

```
ФОНОВАЯ ПРОВЕРКА (каждый час):

FOR EACH task WHERE status = 'awaiting_data':
   blocked_hours = (NOW() - blocked_since).hours
   
   IF blocked_hours > 24:
      → Alert Lead'у (IN-APP):
        "⚠️ Ваша задача ждёт данные > 24 часов"
        [Посмотреть зависимость]
        
   IF blocked_hours > 48:
      → Alert ГИПу (Telegram + IN-APP):
        "⚠️ БЛОКИРОВКА > 48ч: [основная] ждёт [зависимая]"
        "Зависимая статус: [STATUS]"
        [Посмотреть детали]
        
   IF blocked_hours > 72:
      → 🚨 CRITICAL ALERT ГИПу:
        "КРИТИЧЕСКАЯ БЛОКИРОВКА: [задача] заблокирована 72+ часа!"
        "Требуется немедленное действие"
        [Разблокировать вручную] [Посмотреть]
        
   IF blocked_hours > 168 (неделя):
      → Daily ALERT ГИПу
        "Задача заблокирована НЕДЕЛЮ!"
```

### 4.4 ГИП может разблокировать вручную

```
ДЕЙСТВИЕ: ГИП видит заблокированную задачу в Dashboard

СЦЕНАРИЙ: Данные пришли другим способом (почта, вручную и т.д.)

→ ГИП кликает: [РАЗБЛОКИРОВАТЬ ВРУЧНУЮ]
→ Откроется форма:
   Комментарий: "Получены данные по почте от [источник]"
   [ПОДТВЕРДИТЬ РАЗБЛОКИРОВКУ]

РЕЗУЛЬТАТ:
→ Статус = "in_progress"
→ Записать: unblocked_by_gip_at = NOW()
→ История: "[ГИП] разблокировал вручную"
→ Уведомление инженеру: "Данные готовы, можно работать"
```

## 5. КОНТРОЛЬ ДЕДЛАЙНОВ (DEADLINE LOGIC)

### 5.1 Мониторинг дедлайнов

```
ФОНОВАЯ ПРОВЕРКА (каждый час):

FOR EACH task WHERE status != 'approved':
   time_to_deadline = deadline_at - NOW()
   
   IF time_to_deadline <= 2 days AND time_to_deadline > 1 day:
      → status_color = YELLOW
      → ⏱️ Показать оранжевые часы на карточке
      
   IF time_to_deadline <= 1 day AND time_to_deadline > 0:
      → status_color = RED
      → 🔴 СРОЧНО
      → Отправить напоминание (если ещё не отправлено):
        Инженер + Lead (IN-APP):
        "🚨 СРОЧНО: Дедлайн [TASK] завтра в [TIME]"
        
   IF time_to_deadline <= 0:
      → status_color = BLACK
      → ❌ ЗАДАЧА ЗАПОЗДАЛА
      → Отправить alert ГИПу (Telegram):
        "❌ ПРОСРОЧКА: [TASK] (дедлайн был [DATE])"
        "Статус: [CURRENT_STATUS]"
        
   IF status = 'awaiting_data' AND time_to_deadline <= 1 day:
      → CRITICAL для Lead'а:
        "⚠️ СРОЧНАЯ БЛОКИРОВКА: [задача] ждёт данные, дедлайн завтра!"
        "Зависимая задача: [NAME], статус: [STATUS]"
        [СРОЧНО ПОСМОТРЕТЬ]
```

### 5.2 Эскалация по дедлайну

```
ДО ДЕДЛАЙНА 2 ДНЕЙ:
─────────────────
→ Уведомление инженеру: "Осталось 2 дня"
→ Lead видит в очереди: 🟡 ЖЁЛТЫЙ флаг

ДО ДЕДЛАЙНА 1 ДНОГО:
──────────────────
→ Уведомление инженеру (Telegram): "СРОЧНО! Осталось 1 день"
→ Уведомление Lead (Telegram): "Инженер [NAME] спешит, осталось 1 день"
→ ГИП видит в Dashboard: 🔴 КРАСНЫЙ флаг

ДЕДЛАЙН ПРОШЁЛ:
──────────────
→ Alert ГИПу (Telegram): "❌ ПРОСРОЧКА: [TASK]"
→ Dashboard: чёрный флаг, задача в "запоздалых"
→ История: отметить запоздание
→ ГИП может принять решение: расширить дедлайн или вернуть на доработку
```

## 6. ЭСКАЛАЦИЯ И АЛЕРТЫ

### 6.1 Лестница эскалации

```
ПРОБЛЕМА: Lead не проверяет задачу более 24 часов
─────────────────────────────────────────────────

ЧАС 24:
  → Напоминание Lead'у (IN-APP):
    "📦 Вы имеете [N] задач на проверке уже 24+ часа"
    "Пожалуйста, проверьте"

ЧАС 36:
  → Alert Lead'у (Telegram):
    "⚠️ [TASK] ждёт 1.5 дней"

ЧАС 48:
  → Alert ГИПу (Telegram):
    "⚠️ Lead [NAME] не проверяет [TASK] 48 часов"
    "Статус: [STATUS], инженер: [ENGINEER]"
    [ПОСМОТРЕТЬ]

ПОЧТИ ДЕДЛАЙН (< 1 дня):
  → CRITICAL Alert ГИПу:
    "🚨 СРОЧНАЯ ПРОВЕРКА: [TASK] ждёт Lead'а, дедлайн [TIME]!"
    [РАЗБЛОКИРОВАТЬ ВРУЧНУЮ] или [СВЯЗАТЬСЯ С LEAD]
```

```
ПРОБЛЕМА: Инженер не работает 48 часов, задача в "В работе"
──────────────────────────────────────────────────────────

ЧАС 48:
  → Уведомление Lead'у (IN-APP):
    "Инженер [NAME] не трогал [TASK] 2 дня"

ЧАС 72:
  → Alert ГИПу (Telegram):
    "⚠️ Инженер [NAME] неактивен в [TASK] 3 дня"
    "Дедлайн: [DATE]"
    [ПЕРЕНАЗНАЧИТЬ] или [СВЯЗАТЬСЯ]
```

## 7. АВТОМАТИЧЕСКИЕ ДЕЙСТВИЯ

### 7.1 Auto-unblock при утверждении зависимой задачи

```
КОГДА: task.status CHANGED TO 'approved'

1. Проверить: есть ли task_dependencies с dependent_id = этой задаче?

2. Если ДА, для каждой parent_id:
   IF parent.status = 'awaiting_data'
      AND зависимость.resolved_at IS NULL
   THEN:
      → UPDATE parent.status = 'in_progress'
      → UPDATE parent.unblocked_at = NOW()
      → UPDATE dependency.resolved_at = NOW()
      → INSERT в history: "[System] автоматически разблокировала"
      
3. Отправить уведомление инженеру основной задачи

РЕЗУЛЬТАТ: Инженер может сразу продолжить работу
```

### 7.2 Auto-notify на приближение дедлайна

```
КАЖДЫЙ ЧАС (heartbeat):

FOR EACH task WHERE status NOT IN ('approved', 'archived'):
   IF deadline_at - NOW() == 2 days:
      → Отправить IN-APP уведомление всем заинтересованным
      → Выделить на UI жёлтым
      
   IF deadline_at - NOW() == 1 day:
      → Отправить Telegram инженеру + Lead
      → Выделить на UI красным
      
   IF deadline_at < NOW():
      → Alert ГИПу о просрочке
      → Отметить в истории
      → Добавить в Dashboard "Запоздалые задачи"
```

### 7.3 Auto-color на UI

## 9. МЕТРИКИ И МОНИТОРИНГ

### 9.1 Метрики качества

```
БАЗОВЫЕ МЕТРИКИ:

1. Среднее время выполнения задачи
   = (approved_at - created_at) в часах/днях
   Стандарт: < 5 дней

2. Количество возвратов на доработку
   = COUNT(возвратов) / COUNT(утверждённых)
   Стандарт: < 0.3 (30%)

3. Среднее время на проверку (Lead)
   = (submitted_to_gip_at - submitted_to_lead_at)
   Стандарт: < 24 часов

4. Среднее время на проверку (ГИП)
   = (approved_at - submitted_to_gip_at)
   Стандарт: < 12 часов

5. Процент запоздалых задач
   = COUNT(deadline < NOW) / COUNT(всех)
   Стандарт: 0-5%
```

### 9.2 Метрики по отделам

```
СТАТИСТИКА ПО ОТДЕЛУ:

• Всего задач
• Завершено / В процессе / Утверждено
• Среднее число возвратов на инженера
• Среднее время на проверку Lead'ом
• Процент запоздалых
• Сложные задачи (3+ возвратов)

РЕЙТИНГ ИНЖЕНЕРОВ:

• На выполнение (быстро ли делает)
• На качество (сколько возвратов)
• На скорость отправки (как быстро отправляет на проверку)
• На дедлайны (соблюдает ли сроки)
```

## 10. ТАБЛИЦА СОСТОЯНИЙ (STATE MACHINE)

### 10.1 Полная таблица переходов

```
СТАТУС                    | МОЖНО ПЕРЕЙТИ В        | АВТОМАТИЧЕСКИ | УСЛОВИЯ
──────────────────────────────────────────────────────────────────────────────
created                   | in_progress            | -              | ГИП или инженер
                          | awaiting_data          | ДА (если зав) | Есть зависимость
                          
in_progress               | review_lead            | -              | Файл прикреплён
                          | awaiting_data          | -              | ГИП заблокировал
                          
review_lead               | review_gip             | -              | Lead одобрил
                          | rework                 | -              | Lead вернул
                          | in_progress            | -              | Инженер откатил
                          
review_gip                | approved               | -              | ГИП утвердил
                          | rework                 | -              | ГИП вернул
                          
rework                    | review_lead            | -              | Инженер переотправил
                          | in_progress            | -              | Инженер откатил
                          
awaiting_data             | in_progress            | ДА             | Зависимость разрешена
                          | in_progress            | -              | ГИП разблокировал
                          
approved                  | archived               | -              | Ручно или автоматически
                          
archived                  | -                      | -              | Финальный статус
```

### 10.2 Правила валидации переходов

```
НИКОГДА НЕ МОЖЕТ ПЕРЕЙТИ:

• из "approved" в любой другой (кроме archive)
• из "review_*" прямо в другой review_* (только через rework)
• если нет файла из "in_progress" в "review_lead"
• если есть blocker замечание из "review_*" в "review_gip"
• если assignee_id == NULL

ВСЕГДА ТРЕБУЕТСЯ:

• Причина при возврате (минимум одно замечание)
• Файл при отправке на проверку
• ГИП при создании задачи (assigned_by_gip_id)
```

4. UI переинициализирует компонент с новыми данными

REAL-TIME ОБНОВЛЕНИЯ:

• Статус задачи (мгновенно)
• Замечания (при добавлении)
• Уведомления (при событии)
